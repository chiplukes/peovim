# peovim Codebase Map

A guided tour from editor startup to plugin execution. Each entry links to a
`# cm:XXXXXX` anchor embedded in the source. Use your editor's search
(`/cm:ID`) or grep to jump straight to the annotated location.

## 2. Core Data Structures

- [class EditorState](cm://7b9e5f) — Global mutable singleton: active tab, theme, message, decorations
- [class EventBus](cm://4c8a1d) — Pub/sub backbone; plugins subscribe via `api.events.on()`
- [class RegisterStore](cm://f3d2c6) — Named-register storage (yank, delete, macro, etc.)
- [class JumpList](cm://6e4b9a) — Ctrl-O/Ctrl-I jump history per window
- [class Window](cm://8f2d5b) — Viewport + cursor state; wraps a Document
- [class PieceTable](cm://1c7a4e) — O(log n) rope-based text storage
- [class Document](cm://d5f3b8) — Wraps PieceTable; exposes line-based API, folds, marks, filetype
- [class UndoStack](cm://3a6e9c) — Per-document undo/redo; snapshots PieceTable states
- [class Workspace](cm://d4f6b3) — Owns all open documents, tabs, and the window split tree

---

## 3. Modal Engine — Input → Actions

- [class Mode](cm://b2d4f1) — Enum of all editor modes (normal, insert, visual, etc.)
- [class ModalEngine](cm://5c8e7a) — Trie-based FSM: `feed_key()` → `list[Action]`
- [class Action](cm://9f1b3d) — Frozen dataclass base; every editor operation is an Action subclass
- [class PluginContext](cm://4e2c6f) — Snapshot of visual mode, count, register passed to plugin callbacks
- [class ActionDispatcher](cm://7a5d8b) — Routes Actions to Document/Window; owns dot-repeat and insert session
- [dispatch()](cm://1e3f2c) — Main dispatch loop called by EventLoop after each key

---

## 4. Motions & Text Objects

- [move_h() — first motion](cm://c6b4a9) — Character-level movement; all motions return (line, col) tuples
- [_word_range() — iw/aw](cm://2f7d5e) — Text-object range functions returning (sl, sc, el, ec) tuples
- [class BindingRegistry](cm://8c3a1f) — Maps user key sequences to Actions or plugin callbacks

## 5. Async Event Loop — The Main Render Cycle

- [class EventLoop](cm://e4d6b5) — Asyncio main loop: input → engine → dispatch → render at 60 fps
- [run()](cm://6f9a2d) — Entry point: enters raw mode, starts render loop, drives input
- [class EventLoopRuntimeController](cm://b1c5e8) — Runtime maintenance: autosave ticks, yank-flash expiry, dirty flushing

## 6. Rendering — CellGrid & Backend

- [CellGrid import dispatch](cm://3d7f4a) — Loads native Cython extension; falls back to pure-Python CellGrid
- [class TerminalBackend](cm://f8e1c2) — Protocol defining the terminal abstraction (read_events, write, flush)
- [create_backend()](cm://5b4a9e) — Selects prompt_toolkit / crossterm / headless backend at startup
- [class WindowRenderController](cm://9d6c3f) — Collects and dispatches per-window render jobs (sequential or parallel)
- [compute_layout()](cm://2a8b7e) — Recursively assigns Rect to each WindowLeaf in the split tree

## Architecture Summary

```
asyncio.run(event_loop.run())
  │
  ├─ TerminalBackend.read_events()  ← input from terminal
  │     │
  │     ▼
  │  ModalEngine.feed_key()         # cm:5c8e7a  Trie FSM → list[Action]
  │     │
  │     ▼
  │  ActionDispatcher.dispatch()    # cm:1e3f2c  Mutations on Document/Window
  │     │
  │     ├─ Document (PieceTable)    # cm:d5f3b8  Text storage
  │     ├─ Window (cursor/scroll)   # cm:8f2d5b
  │     ├─ RegisterStore            # cm:f3d2c6
  │     └─ EventBus.emit()          # cm:4c8a1d  → plugin subscribers
  │
  └─ _render_loop() at 60fps
        │
        ├─ WindowRenderController   # cm:9d6c3f  parallel render jobs
        ├─ SyntaxEngine (thread)    # cm:5a3c7e
        ├─ SidebarHost / BottomPanelHost
        └─ TerminalBackend.write()  → CellGrid → terminal
```

**Layer relationships:**
- `core/` — pure data, no I/O, no asyncio
- `modal/` — pure functions + FSM, no I/O; depends on `core/`
- `commands/` — ex-command parsing + dispatch; depends on `core/` + `modal/`
- `lsp/` + `syntax/` — background threads, post results to main loop
- `ui/` — asyncio + rendering; orchestrates all layers
- `api/` — thin wrappers over `core/` + `ui/` exposed to `plugins/`
- `config/` — exec's user `init.py` into `api/` namespace
- `plugins/` — pure consumers of `api/`; never imported by core
